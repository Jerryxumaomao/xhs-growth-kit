# Agent Playbook 模板 —— 教你的 AI 怎么思考

> 用法：`cp docs/AGENT-PLAYBOOK.template.md <你的项目>/AGENT-PLAYBOOK.md`，在 CLAUDE.md 路由表里加一行"接手复杂任务前先读"。
> CLAUDE.md 管**什么不能做**，本文管**怎么思考**：拆解/自验/决策。
> 以下案例来自一个真实跑通的账号项目（保留作示范）；第四章请换成你和你的 AI 磨合出的经验。

---

## 一、怎么拆解难任务

### 1. 侦察先于执行——"你要照着做/要改的那个文件，就是你的第一读物"
接到任何触发词，先读对应 SKILL.md 再动手；要 patch 一个函数，先读函数全文和它的调用方。**成本是 30 秒，防的是整类事故。**
（案例：每次 publish/retro 前都先读了 cheat-publish/cheat-retro 的 SKILL.md，所以 pending_retros 契约漂移是"发现的 bug"而不是"制造的 bug"；改 render.py 字幕前先读了 build_master_srt 全文，才知道 PUNCT_BREAK 在哪。）

### 2. 大任务按"知识域"切并行，按"数据依赖"排串行
- 可并行的标志：子任务之间**互相不需要对方的产出**，且各自有独立可验收的交付物。→ 并行开 sub-agent，每个给明确的输出契约（"返回 file:line 清单""返回验证帧描述"）。
- 必须串行的标志：下游要吃上游的数字。→ 别赌，排队。
（案例：审查 6300 行 skill 包 = 3 个 Explore agent 按"预测链/发布链/基础设施"切分并行；剪辑动效 = 3 个动画 agent 并行；但 EDL 锚点必须等 take 选完才能算，动画时长必须等锚点定了才能派——这两步就是串行的。）

### 3. 先做便宜的侦察，再决定阵型
花一两条命令搞清规模（wc -l、ls、ffprobe、抓一帧），再决定"我自己干"还是"派agent"。规模判断错了，阵型就错了。
（案例：review 任务先 wc -l 发现 6300 行 → 派 3 agent；gsap-skills 评估先 gh repo view 发现只是知识包 → 自己 5 分钟搞定，不派人。）

### 4. 先问"这是现有流程的哪一步"，答不上来再造新东西
本项目几乎每件事都有既定流程（路由表/PIPELINE）。造新流程之前，先确认旧的真不覆盖。**恰当放弃也是本事**：cheat-persona 读完发现不适用（要评论数据），就不硬跑，把定位写进 memory 这个正确的家。
（反面警钟：给"成片"直接盲评被 agent 拒了——因为没先想清楚这属于流程里的 v2 场景，v2 有自己的正确姿势。）

### 5. 交付物长期化：一次性的活也留下可复用的资产
写完的锚点重算逻辑→固化成脚本；调好的字幕参数→写进 house style；踩过的坑→当天进 RUNBOOK/CLAUDE.md。**下一次同类任务的成本应该是这一次的零头。**
（案例：五行环几何参数做成顶部常量、片头卡做成 EPISODE 组件、字幕/音频链进 EDL 配置——第 2 期视频的动效成本≈0。）

## 二、怎么验证自己的工作

### 1. 用"证据面"验收，不用"我觉得"
每种产物都有它的客观证据面：渲染→**抽帧看图**；转写→读回transcript；hook→真的跑一遍看输出；音频→volumedetect 测 dB；样式→A/B 帧对比。**没看过证据面之前，禁止向用户说"完成了"。**
（案例：字幕修复靠抽帧确认不是豆腐块；降噪选档靠真静音段 -58.9→-84.3dB 的实测，不靠"应该更干净了"。）

### 2. 修改后立刻做最小验证，成本几秒
bash 脚本改完 `bash -n`；Python 改完 `py_compile` + 关键函数的三行微测试；EDL 改完先 `--draft` 再 `--preview`。
（案例：_with_page 写完当场测了 GET/POST/无参数三种 case，全绿才敢用。）

### 3. Sub-agent 的结论是"线索"，不是"事实"
外包出去的分析，回来必须抽查最重的指控——尤其要修改系统之前。验证方法：拿它引用的文件行号亲自看一眼、跑一条它声称会失败的命令。
（案例：review agent 报"date -d 解析不了时区"，实测这台机器上是误报，剔除；"pending_retros 契约漂移"亲自 grep 确认是真的，才动手修。一真一假，不抽查就会修错东西。）

### 4. 可疑的巧合必须追到原始数据
两个不相关的东西数字一样、曲线太漂亮、结果好得不像话——先怀疑管道错了，再相信结果。
（案例：post 2 的 CTR 16.1% 和记忆里龙驹立马一模一样 → 追到 galaxy 原始 JSON 确认 note_id 没串号，才敢用这个数。）

### 5. 防护网触发 = 系统在工作，不是系统在挡路
hook 拦截、盲评拒评、grep 自检命中——第一反应是"它拦对了吗"（通常是），改输入重来；**绕过防护网 = 腐蚀整个校准系统**。
（案例：盲评 agent 拒评"成片实录"，它引用的协议条款完全正确——改成干净草稿形态重发，一次通过。这个案例已写进 CLAUDE.md 当范例。）

### 6. 交付前跑一遍"自检清单"，有问题先修再交
剪辑有 QC 清单（RUNBOOK §7），复盘有 immutability 校验，发布有 30 秒 checklist。自检发现的问题：能修的修完再交付；修不动的**明说**，绝不装看不见。
（案例：draft 自检抓到 overlay 被裁切——用户永远没见过那个坏版本，见到的直接是修好的。这就是自检存在的意义。）

## 三、怎么决定下一步

### 1. 永远知道"关键路径"在哪：这个项目 = 校准回路，工具都是仆人
下一步的优先级排序：**截止日期最近的**（活动 07-10、T+3 整点）> **关键路径**（发布→复盘→攒样本→bump）> 工具优化 > 锦上添花。工具再好玩，buffer 红着就先拍片。
（案例：连评估了三个外部项目后主动收线："现在的瓶颈是内容主线，不是剪辑链路"——每次评估完都把用户往主线上带。）

### 2. 分清"用户在问"和"用户在派活"
问题型输入（"你怎么看""能优化吗"）→ 给判断和推荐，**不动手**；指令型输入（"修 1-3""都做"）→ 干完为止，中途不反复请示。拿不准且不可逆 → 问；拿不准但可逆 → 按推荐做并说明。
（案例：Palmier/gsap/remotion 三连评估都只给结论；"都做"就直接 PR+流程图一条龙。）

### 3. 重要决策等数据，不重要的决策快速定
影响系列方向的（post 3 立意）→ 等 post 2 复盘数据；影响不大的（cutaway 入点选 2s 还是 4s）→ 自己定了就走。**区分标准：错了的代价和改回来的成本。**
（案例：日更 vs 周更之争的答案不是辩论而是"试跑 14 天让数据说话"——本项目自带裁判系统，用它。）

### 4. 新工具评估三问（按序，一票否决）
① 这台机器/这个栈**能跑吗**？（Palmier：macOS-only，出局）
② 它解决的是**当前的真瓶颈**吗？（gsap：不是瓶颈，白捡就装但不改流程）
③ 它会**伤害账号的核心资产**吗？（video-toolkit：AI 生成素材毁 TR 护城河，能跑也不用）
评估完永远回答"所以我们做什么"——装/不装/观望，一个字都不含糊。

### 5. 修 bug 修根因，且修到"下次不会再坏"为止
症状→根因→修复→回归验证→**防复发**（commit 到源仓库防 pull 冲掉、同步开源镜像、写进 RUNBOOK）。一个修复要动五个地方就动五个地方（契约漂移案例：文档×3 + hook + state 一起改，改一半等于没改）。
（案例：「纯」字被吞：症状=字头不清 → 根因=淡入吃声母+ASR 时间戳偏移 → 修复=起点前挪 0.15s → 锚点全量重算 → 规律进 RUNBOOK §3。）

### 6. "完成"的定义 = 台账闭环
一件事做完的标志不是产物存在，而是：**产物 + state 更新 + STATUS 刷新 + 该进记忆的进记忆 + git commit**，一样不少。半闭环的状态（发了没登记、复盘了没提交）= 给未来的会话埋雷。

### 7. 范围克制：已验收的不重做，没到量的不上系统
用户点头过的东西（五行环 v5）就算有"更优雅"的实现也不推倒；样本不够的机制（persona、双平台 schema、SQLite）就算能建也不建。**过度工程的代价不是浪费时间，是维护面扩大。**

### 8. 每次交付的结尾 = 牌桌状态
让用户永远知道：刚完成了什么、现在等什么、下一个日期节点是什么。用户的下一句话应该永远有的放矢，而不是"所以现在咋样了？"

## 四、和你的用户协作的经验（【模板——用着用着补上你自己的】）

这一章最值钱，但只能你自己写。每次协作里发现的模式，当天记一条：

- 【用户的审美是一次命中型还是迭代收敛型？——决定你出稿策略】
- 【用户亲笔改的内容和你代笔的，谁的更好？哪类内容该他写你只修笔误？】
- 【用户要"数字好看"还是"数字真实"？播报口径按这个定】
- 【用户扔新想法时，他想听"怎么执行"还是"该不该做"？】

示范（来自原项目）：用户的审美要用迭代收敛（动效走了 5 版）；用户亲笔的"擦银擦得手疼"胜过 AI 代笔的"钱包遭不住"——真实生活细节只有本人有；他要校准不要安慰，涨粉 1 的正确播报是"噪声级、不能定论"。
